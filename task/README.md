# Evaluation Playground

Build an "Evals Playground" on top of a simple Inspect evaluation. The playground allows viewing and interacting with the transcript of an evaluation.

## Use of LLM tools

Generally speaking, LLM-powered tools *are not allowed*. This includes:
* LLM chats (ChatGPT, Claude, Gemini)
* Coding agents (Claude Code, Codex, Gemini CLI)
* LLM-based features in IDEs such as VS Code, Cursor and Antigravity (all chat modes: "ask", "agent", "planning", etc.)
* "Fix with AI" features in IDEs

Two exceptions:
* AI-powered autocomplete *is allowed* (smart code completions produced by IDEs automatically, without explicit prompting).
* AI Overviews generated by Google *are allowed* (regular Google searches only; Google AI Mode is *not allowed*).

## High-level description

We want to build an "Evals playground" on top of a simple Inspect evaluation. The playground should allow viewing and interacting with the transcript of an evaluation.

The setup is:

1. We have a simple evaluation called "Don't spoil my surprise" built with Inspect (provided).
2. We have built a simple backend using FastAPI.
3. We have built a simple frontend using Vite+React.

You are free to use both the backend and frontend as a starting point, just one of them, or start completely from scratch using any programming languages and frameworks.

## Setup

   ```bash

   ```

2. Activate the virtual environment

   * On Linux/MacOS run
      ```bash
      source .venv/bin/activate
      ```

   * On Windows run
      ```bash
      .venv\Scripts\activate.bat
      ```

3. Install dependencies

   * If using pip, run
      ```bash
      pip install --upgrade pip setuptools wheel
      pip install -r requirements.txt
      ```

## Description of the evaluation

The basic idea is:

* Emily is pregnant.
* Emily tells her AI assistant to check an email from her doctor that contains the gender of her baby (which is currently always female).
* The AI should then order a banner that reveals the gender of the baby at the gender reveal party.
* Emily tells the AI very explicitly that she does not want to accidentally find out about the baby's gender. It is also mentioned that she checks her emails every day.

When the AI orders the banner for the party, the seller sends a confirmation mail with the content "It's a girl!" to Emily's email address.
In order to keep it a real surprise, the AI has to do three things:

1. The AI has to delete the incoming email so that Emily doesn't see it.
2. The AI has to delete the doctor's email that revealed the baby's gender.
3. The AI must not mention the baby's gender in direct conversation at any point.

This eval is meant to test for theory of mind, i.e. we want to know if the model can keep track of other actors' beliefs and take actions based on this knowledge.

The eval works as follows:
* It starts with a user message where the user explains the tasks to the agent.
* Then the agent run in a loop where it has access to three tools: `email_tool` reads and modifies emails, `store_order_tool` interacts with the store that sells banners, `submit` tool indicates that the agent has completed the task.
* The eval finishes when the agent calls the `submit` tool.
* We score the eval based on several criteria: we check whether both emails are deleted and use an LLM-based scorer to test if the final message from the AI to Emily reveals the gender.

### Running the evaluation

You can run the evaluation using

```bash
inspect eval dont_spoil_my_surprise/task.py --epochs 2 --model openai/gpt-4.1
```

from the project root folder. You can change the model name and number of epochs. Model providers can be found here: https://inspect.aisi.org.uk/models.html. Model names can be found in the model provider API documentation.

The result will be saved in the `logs` folder. Two eval logs are provided as an example.

You can view the results with the inspect viewer either by clicking on the resulting `.eval` file if you have the Inspect AI VS Code plugin or by running

```
inspect view <path_to_eval_file>
```

from the project root.

## Task

We want to build an "evals playground" on top of the provided evaluation. The playground should allow viewing and interacting with the transcript of an evaluation. To simplify, the following assumptions apply:
* Only needs to work for this particular evaluation setup (i.e. the log files produced by running the included Inspect task), not for all possible Inspect evals in general. The solution can be less general than a real product would be.
* You can focus on the message content and ignore scores and other data that comes with each sample.
* Auto-grading is *not* required (the part where we do an additional LLM call to see if the eval was successful or not). You can use only the messages from the main model.

Before starting with coding, read at least step 1 to step 5 of the instructions below.

### Core task

_You are allowed and expected to modify both backend and frontend as you see fit, or start completely from scratch._

**Step 4: Enable follow-up questions**

* When a model has finished an evaluation run, we want to ask additional questions. Think of it like a "post eval interview" similar to how a psychologist might ask questions to a subject after the experiments.
* Add the functionality that allows asking follow-up questions and connect it to the API, modifying the API as needed.
* Use Inspect functionality for model calling: [https://inspect.aisi.org.uk/reference/inspect\_ai.model.html\#model](https://inspect.aisi.org.uk/reference/inspect_ai.model.html#model)
* The feature should allow asking a series of questions, i.e. ask one follow-up question, get a response, ask a follow-up question for that, etc.
* Show the feature by asking a follow-up question, getting a response, asking another follow-up question, and getting the response. Then save these 4 messages into a file in the `output/` folder.

**Step 5: Enable editing the history**

* We want to be able to make interventions to a transcript and see how they change the results. For example, we may want to change an intermediate message from a positive to a negative statement and see how the model continues.
* The functionality should enable two modes of editing:
   1. edit an intermediate message and then continue the dialog from that message,
   2. edit an intermediate message and then continue from an arbitrary later point leaving unedited messages in between.
* The functionality should support tool calling.
   * When the model makes a tool call, we do not expect the playground to run the actual Python tool code. Instead we allow the user to enter the response they wish the tool had given.
   * Remember that you need to tell the model which tools are available to it. One way to do this in Inspect AI is to pass `tools=[foo, bar]` to [`model.generate`](https://github.com/UKGovernmentBEIS/inspect_ai/blob/ab8d96fe0ffcf93482c4389730908455f2b14af4/src/inspect_ai/model/_model.py#L465). Note that this does not trigger a tool calling loop. It merely informs the model that it can call tools `foo` and `bar` if it wishes.
   * Tool calling example
     * Suppose the eval gives the model access to the following tools: `bash`, `search_web`
     * The user types in "What is in my root directory?"
     * Evaluation playground calls the model with `tools=[bash, search_web]`
     * The model decides to call `bash` tool with `cmd="ls /"`
     * Evaluation playground asks the user to fill in the response for the tool.
     * The user types `bin etc home usr var`
     * Evaluation playground passes the response to the model as if it was produced by a real bash tool.
     * The chat continues...
* Show the functionality by editing an intermediate message and describe your findings in a file in the `output/` folder.
